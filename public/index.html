PeerBasket

A hassle-free, lobby-based PeerJS discovery server.

Star on GitHub ·

How It Works

PeerBasket is a public, zero-config room discovery API for PeerJS. A single API route registers you to any arbitrary lobby and returns every other peer currently active in that same lobby — no signaling infrastructure of your own required. A hosted instance lives at peerbasket.bittu.dev, or self-host it yourself from the GitHub repo.

Why does this Exist?

PeerJS solves the connection, but not the introduction: it can open a P2P link once both sides already know each other's peer ID, but it has no notion of rooms or discovery. PeerBasket fills that one gap. Built for developers who just want to deploy peer-to-peer networks without standing up dedicated infrastructure, it streamlines swarm discovery and intentionally avoids authentication by design — see Security Considerations below for what that trade-off actually means. This service is completely free and open source — if it's useful to you, consider supporting it on Patreon.

API Documentation

POST peerbasket.bittu.dev/basket/:id?limit=100
limit optional, default: 100

The single API endpoint. It expects a required JSON body containing the client's Peer ID. It registers the peer into the lobby and returns a list of active peers from the same lobby.

The <id> (the "basket ID") can be any arbitrary string. Only clients using the same basket ID can discover each other — there's no separate room-creation step.

Required JSON Body

{
  "peer_id": "peerjs-abc-123"
}

Response Payload 

{
  "basket_id": "room-101",
  "peers": ["peerjs-id-001", "peerjs-id-002"],
  "total_peers": 2,
  "peers_returned": 2
}
  • • A peer that hasn't made a request in the last 30 seconds is considered inactive and automatically dropped from the lobby.
  • • Rate Limiting: The edge network enforces a hard rate limit of 4 requests per 10 seconds per IP address. Exceeding this boundary will result in an HTTP 429 error. Ensure your application architecture leverages a polling interval of 10 seconds or slower.
  • • The returned peer list includes your own peer ID alongside everyone else's — filter it out client-side before connecting to avoid trying to connect to yourself.
  • • The peer list reflects each peer's last heartbeat, not whether it's online right now — a peer that crashed or lost its connection can still appear active for up to 30 seconds. Treat a successful PeerJS connection as the real liveness check, and handle a connection that never opens or errors out, rather than assuming every returned peer is reachable.
  • • There's no way to reserve a basket ID — anyone who knows it can join and see its peer listings.
  • • The value of peers_returned may be less than total_peers if the requested query limit value is smaller than the absolute lobby size.

Example Usage

Here's the API above put into practice: spin up a PeerJS peer, then register it into a lobby and poll for company. A 10-second polling interval is recommended to stay safely inside the network's rate limit bounds.

1. Create a peer

// <script src="https://unpkg.com/peerjs@1.5.4/dist/peerjs.min.js"></script>

const peer = new Peer(); // PeerJS assigns a random ID via its own broker server
const connections = new Map(); // peerId -> open DataConnection

peer.on('open', (id) => {
  console.log('My peer ID is:', id);
  joinLobby();
});

peer.on('connection', (conn) => {
  connections.set(conn.peer, conn);
  conn.on('data', (data) => console.log('Received:', data));
});

2. Join a basket & connect to peers

async function joinLobby() {
  const res = await fetch('https://peerbasket.bittu.dev/basket/room-101', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ peer_id: peer.id })
  });

  // Handle network rate limits gracefully
  if (res.status === 429) {
    console.warn('Rate limited by edge gateway. Retrying on next interval...');
    return;
  }

  const { peers } = await res.json();

  peers
    // Skip yourself (the list includes your own ID) and anyone
    // you're already connected to, so reconnects don't pile up.
    .filter((id) => id !== peer.id && !connections.has(id))
    .forEach((id) => {
      const conn = peer.connect(id);
      connections.set(id, conn);

      // A peer can still show up as "active" for up to the heartbeat
      // window after it's actually gone — don't assume it's reachable
      // until the connection actually opens.
      const giveUp = setTimeout(() => {
        connections.delete(id);
        conn.close();
      }, 5000);

      conn.on('open', () => {
        clearTimeout(giveUp);
        conn.send('hello!');
      });

      conn.on('error', () => {
        clearTimeout(giveUp);
        connections.delete(id);
      });
    });
}

// Re-register periodically to stay "active" and catch newcomers
// Kept at 10 seconds to keep your shared client window clear
setInterval(joinLobby, 10_000);

Security Considerations

PeerBasket does no authentication, by design — a basket ID is the only thing standing between a lobby and the public. Treat a basket ID like an unlisted URL, not a password, and pick something unguessable for anything that isn't a throwaway demo.

This also means PeerBasket can't vouch for who's behind any peer ID it returns. Anyone who knows your basket ID can register their own peer into it, and your app will see that peer as just another lobby member. If you're building anything beyond a quick prototype, verify identity at the application layer — a shared secret exchanged out-of-band, or a handshake once the PeerJS connection actually opens — rather than trusting the raw peer list on its own.
