PeerBasket

A hassle-free, lobby-based PeerJS discovery server.

Star on GitHub ·

How It Works

PeerBasket is a public, zero-config room discovery API for PeerJS. A single API route registers you to any arbitrary lobby and returns every other peer currently active in that same lobby — no signaling infrastructure of your own required. A hosted instance lives at peerbasket.bittu.dev, or self-host it yourself from the GitHub repo.

Why does this Exist?

PeerJS solves the connection, but not the introduction: it can open a P2P link once both sides already know each other's peer ID, but it has no notion of rooms or discovery. PeerBasket fills that one gap. Built for developers who just want to deploy peer-to-peer networks without standing up dedicated infrastructure, it streamlines swarm discovery and intentionally avoids authentication by design — see Security Considerations below for what that trade-off actually means. This service is completely free and open source.

Quickstart

Once your PeerJS client has an ID, register it into a lobby and poll for company. Anything under the 30-second timeout keeps you active — around 10 seconds is a reasonable default. 

// Register this client and discover peers in "room-101"
async function checkLobby() {
  const res = await fetch('https://peerbasket.bittu.dev/basket/room-101', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ peer_id: myPeer.id })
  });
  const { peers } = await res.json();
  // peers: every other active peer_id in this basket right now
  return peers;
}

// Re-register periodically to stay "active" and catch newcomers
setInterval(checkLobby, 10_000);

API Documentation

POST peerbasket.bittu.dev/basket/:id?limit=100
(limit is optional, default: 100)

The single API endpoint. It expects a required JSON body containing the client's Peer ID. It registers the peer into the lobby and returns a list of active peers from the same lobby.

The <id> (the "basket ID") can be any arbitrary string. Only clients using the same basket ID can discover each other — there's no separate room-creation step.

Required JSON Body:

{
  "peer_id": "peerjs-abc-123"
}

Response Payload:

{
  "basket_id": "room-101",
  "peers": ["peerjs-id-001", "peerjs-id-002"],
  "total_peers": 2,
  "peers_returned": 2
}
  • • A peer that hasn't made a request in the last 30 seconds is considered inactive and dropped from the lobby.
  • • There's no way to reserve a basket ID — anyone who knows it can join and see its peers.
  • • peers_returned may be less than total_peers if limit is smaller than the lobby's size.

Security Considerations

PeerBasket does no authentication, by design — a basket ID is the only thing standing between a lobby and the public. Treat a basket ID like an unlisted URL, not a password, and pick something unguessable for anything that isn't a throwaway demo.

This also means PeerBasket can't vouch for who's behind any peer ID it returns. Anyone who knows your basket ID can register their own peer into it, and your app will see that peer as just another lobby member. If you're building anything beyond a quick prototype, verify identity at the application layer — a shared secret exchanged out-of-band, or a handshake once the PeerJS connection actually opens — rather than trusting the peer list on its own.
